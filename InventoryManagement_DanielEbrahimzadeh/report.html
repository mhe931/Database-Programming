1. Project Context & Motivation

For this second phase of the course, I moved from raw JDBC to using the Java Persistence API (JPA) with Hibernate. As an AI student, I'm usually dealing with high-level data abstractions, so writing manual SQL strings in Java felt a bit clunky. Moving to Object-Relational Mapping (ORM) makes the whole system feel more modern... and easier to work with.

I chose the products table again because it's exactly the kind of structured data we see in inventory optimization. Instead of thinking about tables and primary keys, I can just think about Product objects. It’s a much more natural way to build software... it lets me focus on the logic rather than the database syntax.

2. Technical Architecture

The project uses a Maven-based structure. Getting Hibernate to talk to SQLite was the main task here... since SQLite is a file-based database, the configuration needs to be very precise to avoid file locks.

2.1 Project Dependencies (pom.xml)

Maven Configuration

<dependencies>
    <!-- Hibernate Core -->
    <dependency>
        <groupId>org.hibernate.orm</groupId>
        <artifactId>hibernate-core</artifactId>
        <version>6.4.4.Final</version>
    </dependency>
    <!-- SQLite JDBC Driver -->
    <dependency>
        <groupId>org.xerial</groupId>
        <artifactId>sqlite-jdbc</artifactId>
        <version>3.45.1.0</version>
    </dependency>
</dependencies>

2.2 Persistence Configuration

I placed the persistence.xml in META-INF. I used the community SQLite dialect to ensure Hibernate generates the correct SQL for the SQLite engine.

src/main/resources/META-INF/persistence.xml

<persistence-unit name="InventoryPU">
    <provider>org.hibernate.jpa.HibernatePersistenceProvider</provider>
    <class>com.inventory.jpa.entity.Product</class>
    <properties>
        <property name="jakarta.persistence.jdbc.driver" value="org.sqlite.JDBC"/>
        <property name="jakarta.persistence.jdbc.url" value="jdbc:sqlite:inventory_jpa.db"/>
        <property name="hibernate.dialect" value="org.hibernate.community.dialect.SQLiteDialect"/>
        <property name="hibernate.hbm2ddl.auto" value="update"/>
    </properties>
</persistence-unit>

3. Data Access Implementation

The Product entity acts as the blue-print. I used annotations to tell JPA which fields map to which columns. I really like how @GeneratedValue handles the ID automatically... one less thing for me to manage manually in the code.

Product.java (Entity Model)

@Entity
@Table(name = "products")
public class Product {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    private String name;
    private String category;
    private double price;
    private int stock;

    // Standard getters and setters follow...
}

3.1 The DAO Logic

The ProductDAO handles the EntityManager lifecycle. For a simple app like this, it’s important to open and close managers correctly to keep the SQLite file healthy.

ProductDAO.java (Data Access)

public void addProduct(Product product) {
    EntityManager em = emf.createEntityManager();
    try {
        em.getTransaction().begin();
        em.persist(product);
        em.getTransaction().commit();
    } finally {
        em.close();
    }
}

4. Challenges & Self-Reflection

"Actually, the transition wasn't completely seamless for me. I had some SLF4J logger issues at first... it's a bit annoying when the console is full of warnings. Also, managing transaction commits manually was a bit different from what I expected. But seeing the SQL output generated by Hibernate made me realize how much manual work it's actually saving... specially for the 'update' and 'delete' parts."

One specific thing I learned is that JPA is very strict about transactions. If you forget to commit, the data stays in memory but never actually reaches the inventory_jpa.db file. It took me a moment to realize why my tests were "working" but the database was still empty!
